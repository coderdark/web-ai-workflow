# Grill-Me Session — ApexTuts Website
**Date:** 2026-05-09  
**Source:** idea.md  
**Project:** ApexTuts — AI tutorials/classes platform for non-technical users

---

## Q&A Decision Log

### 1. Homepage Primary CTA
**Q:** What is the primary action a first-time visitor should take when they land on the homepage?  
**A:** See a featured/hero class  
**Implication:** Homepage has a hero section spotlighting one class with a strong CTA ("Start This Class" or "Register"). Class catalog is secondary / below the fold.

---

### 2. Identity & Progress Tracking Without Auth
**Q:** Since there's no auth, how does the app know who is taking a class?  
**A:** Anonymous session via localStorage UUID  
**Implication:** On first visit, generate a UUID and store it in localStorage. All progress is keyed to this UUID. Progress is lost if localStorage is cleared or a different browser/device is used. This is acceptable for the no-auth MVP.

---

### 3. Class Structure
**Q:** What does a 'class' actually contain?  
**A:** Lessons (video + text per lesson)  
**Implication:** Data model is: `Class` has many `Lessons`. Each lesson has ordered content. Progress is tracked at the lesson level.

---

### 4. Video Hosting
**Q:** Where does video content live?  
**A:** No video — text/markdown only for now  
**Implication:** Lessons are markdown-only for MVP. Video support is a future enhancement. Simplifies the lesson viewer significantly.

---

### 5. Registration Flow
**Q:** What does "register for a class" mean without auth?  
**A:** Register = start the class immediately  
**Implication:** Clicking "Register" or "Start Class" on the class detail page sets a `classId` entry in localStorage (status: in_progress) and navigates to lesson 1. No email collected, no backend write needed.

---

### 6. Progress Tracking Mechanic
**Q:** How is progress tracked per lesson? What counts as "completed"?  
**A:** Manual "Mark as Complete" button  
**Implication:** Each lesson view has a prominent "Mark as Complete" button. Clicking it sets `lessonId` as complete in localStorage. No scroll detection, no quizzes needed for MVP.

---

### 7. Progress Storage
**Q:** Where is progress stored?  
**A:** localStorage only (no backend)  
**Implication:** SQLite is **only** used for the class/lesson catalog (read-only from the frontend's perspective). All student state (enrolled classes, completed lessons) lives in the browser's localStorage. Backend is simplified to a read-only catalog API.

---

### 8. Content Management / Seeding
**Q:** How does content get into the database?  
**A:** Seed script / manual SQL inserts  
**Implication:** A developer runs a seed script to populate classes and lessons. No admin UI needed for MVP. This means content updates require a developer touching the DB directly or re-running a seed.

---

### 9. Frontend ↔ Backend Communication
**Q:** How does the React frontend talk to SQLite?  
**A:** Express.js API server with better-sqlite3  
**Implication:** The project has two processes in dev: Vite dev server (frontend, port 5173) and Express server (API, e.g. port 3001). Vite proxies `/api` requests to Express. better-sqlite3 handles synchronous SQLite access server-side.

---

### 10. MVP Routes
**Q:** Which routes are in scope for the MVP?  
**A:** All four core routes selected  

| Route | Description |
|---|---|
| `/` | Homepage — hero with featured class |
| `/classes` | Class catalog — browse all classes |
| `/classes/:id` | Class detail + Register/Start button |
| `/classes/:id/lessons/:lessonId` | Lesson viewer with Mark Complete + prev/next |

---

### 11. Progress Dashboard
**Q:** Should there be a "My Progress" page?  
**A:** Yes — `/progress` page reading from localStorage  
**Implication:** A 5th route is in scope. Reads localStorage to show all enrolled classes and their completion %. No backend call needed.

---

### 12. Visual Design Direction
**Q:** Visual style for the site?  
**A:** Soft neutral — warm off-white + earthy tones  
**Implication:** Tailwind config should use a warm palette: off-white backgrounds (e.g. `stone-50`, `amber-50`), earthy accents (amber, warm brown, sage green). Avoid stark white or dark-mode defaults. Approachable, not sterile.

---

### 13. Lesson Content Format
**Q:** How is lesson content authored and stored?  
**A:** Markdown stored as TEXT in SQLite, rendered with react-markdown  
**Implication:** `lessons.content` column is a TEXT field containing markdown. Frontend uses `react-markdown` (+ optionally `remark-gfm`) to render it. Authors write/edit markdown directly in the seed script or via future tooling.

---

### 14. Class Catalog Card Content
**Q:** What does each class card in the catalog show?  
**A:** Title + short description + lesson count + difficulty badge  
**Implication:** The `classes` table needs: `title`, `description` (short), `difficulty` (enum: Beginner/Intermediate/Advanced). Lesson count is derived via JOIN or a stored count. No thumbnail images needed for MVP.

---

### 15. MVP Definition of Done
**Q:** What must be true before showing this to real students?  
**A:** At least 1 real class with 3+ lessons seeded, all 5 routes working, progress tracked in localStorage  
**Implication:** MVP ships when: (1) catalog, detail, lesson viewer, and progress pages are all functional end-to-end, (2) at least one real ApexTuts class is seeded with 3+ real lessons, (3) a student can start the class, mark lessons complete, and see their progress on `/progress`.

---

## Key Architecture Decisions Summary

```
Frontend:  Vite + React + TailwindCSS + Zustand + React Router
Backend:   Express.js (Node) + better-sqlite3
Database:  SQLite3 (catalog only — classes + lessons)
State:     localStorage (all student progress + session UUID)
Content:   Markdown (stored in SQLite, rendered via react-markdown)
Auth:      None (MVP)
```

## Data Model (Inferred)

```sql
-- Classes table
CREATE TABLE classes (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  title       TEXT NOT NULL,
  description TEXT NOT NULL,       -- short, shown on card
  difficulty  TEXT NOT NULL,       -- 'Beginner' | 'Intermediate' | 'Advanced'
  is_featured INTEGER DEFAULT 0,   -- 1 = show on homepage hero
  created_at  TEXT DEFAULT (datetime('now'))
);

-- Lessons table
CREATE TABLE lessons (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  class_id   INTEGER NOT NULL REFERENCES classes(id),
  title      TEXT NOT NULL,
  content    TEXT NOT NULL,        -- markdown
  order_idx  INTEGER NOT NULL,     -- lesson sequence within the class
  created_at TEXT DEFAULT (datetime('now'))
);
```

## localStorage Schema (Inferred)

```json
{
  "apextuts_session_id": "<uuid-v4>",
  "apextuts_progress": {
    "<classId>": {
      "status": "in_progress | completed",
      "enrolled_at": "<ISO date>",
      "completed_lessons": ["<lessonId>", "<lessonId>"]
    }
  }
}
```

## Routes Summary

| Route | Component | Data Source |
|---|---|---|
| `/` | HomePage | Express `/api/classes?featured=1` |
| `/classes` | ClassCatalog | Express `/api/classes` |
| `/classes/:id` | ClassDetail | Express `/api/classes/:id` + localStorage |
| `/classes/:id/lessons/:lessonId` | LessonViewer | Express `/api/lessons/:lessonId` + localStorage |
| `/progress` | ProgressDashboard | localStorage only |

## Open Questions / Future Considerations

- **Auth phase:** When auth is added (likely Clerk or Supabase Auth), localStorage progress should be migrated to a user record in the DB.
- **Content authoring:** Seed script works for now but will become a bottleneck as the class library grows. An admin UI (even unauthenticated) is a near-term need.
- **Video support:** Designed to be addable — add a `video_url TEXT` column to `lessons` and conditionally render an iframe in LessonViewer.
- **Difficulty filter:** The catalog should support filtering by difficulty badge — Zustand store is a natural fit for this filter state.
- **Featured class rotation:** `is_featured` flag on the class — only one should be featured at a time. Seed script handles this for now.
