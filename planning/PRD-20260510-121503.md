# PRD: ApexTuts v1 Website and Learning Experience

## 1. Project specifics
- Participants: Founder/Product (ApexTuts), Design, Engineering
- Status: Planning complete, implementation-ready
- Target release: Local/private v1 completion by June 30, 2026

## 2. Team goals and business objectives
- Goal 1: Launch a focused website where non-technical users can discover AI classes and enroll quickly.
- Goal 2: Deliver a usable lesson experience with progress tracking in v1 constraints.
- Business objective(s): Validate demand and messaging through conversion performance.
- Success criteria:
  - Visitor-to-registration conversion reaches 8% within 60 days of launch.
  - End-to-end flow works: discover -> class detail -> enroll -> start lesson -> mark complete -> progress persists after refresh (same browser/device).
  - Funnel analytics captured in SQLite for conversion and engagement reporting.

## 3. Background and strategic fit
- Problem/opportunity: ApexTuts has classes/tutorials but no website for discovery, enrollment, and structured lesson consumption.
- Why now: Early company stage requires a low-complexity platform to start learning from real user behavior and conversion quickly.
- Strategic alignment: This v1 supports audience growth through organic social and newsletter traffic and provides measurable proof of value before adding heavier features (auth/payments/admin).

## 4. Assumptions
- Assumption: Primary user segment is non-technical working adults (25–45) seeking practical AI upskilling.
- Validation plan: Track conversion and lesson-start/completion events by segment-oriented landing copy and class topics.
- Risk if wrong: Messaging and class catalog may not resonate; conversion target may be missed.

- Assumption: Registration with only email + first name maximizes conversion for v1.
- Validation plan: Monitor registration drop-off from class detail to enroll success.
- Risk if wrong: Lower lead quality or weak re-engagement.

- Assumption: localStorage-only progress is acceptable for v1.
- Validation plan: Monitor support feedback and lesson completion anomalies.
- Risk if wrong: Trust loss due to unrecoverable progress on browser/device change or cleared data.

- Assumption: Accessibility baseline can be deferred.
- Validation plan: Run practical keyboard/contrast checks during QA even without formal compliance commitment.
- Risk if wrong: Rework and usability barriers for learners.

## 6. User interaction and design
- User flow summary:
  - User lands from organic social/newsletter.
  - User browses catalog (search, topic filters, difficulty tags).
  - User opens class detail and enrolls using first name + email.
  - User starts class and navigates modules/lessons freely.
  - User marks lessons complete manually; progress updates in-app and persists after refresh on same browser/device.
- Wireframes/mockups: Not provided yet; to be created before UI polish pass.
- UX constraints/notes:
  - Visual direction: cool neutrals + teal accent, modern but low-noise.
  - Guardrails: one accent color, max two font families, strict spacing scale, no animated background effects.
  - Demo/placeholder classes must be labeled "Preview Class" and use waitlist CTA (no misleading enrollment).
