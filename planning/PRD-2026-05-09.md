## 2. Team Goals and Business Objectives

- **Goal 1:** Give prospective students a place to discover ApexTuts classes and start learning immediately — no account required.
- **Goal 2:** Track individual student progress at the lesson level so returning students can pick up where they left off.
- **Business Objective:** Establish an owned web presence that converts visitors into active learners and validates the class catalog before investing in auth, payments, or a CMS.
- **Success Criteria (MVP):**
  - At least 1 real class with 3+ real lessons is live and accessible end-to-end
  - All 5 routes are functional (`/`, `/classes`, `/classes/:id`, `/classes/:id/lessons/:lessonId`, `/progress`)
  - A student can enroll in a class, complete lessons, and see their progress on `/progress`
  - Site renders correctly on mobile (375px) and desktop (1280px)

---

## 3. Background and Strategic Fit

**Problem/Opportunity:**
ApexTuts is producing AI education content for non-technical audiences but has no web presence. Students who complete in-person sessions or discover the brand have nowhere to go. Without a site, there is no way to scale content delivery or track engagement.

**Why Now:**
The content library exists (or is being created). The fastest path to proving the model is shipping a lightweight catalog + lesson viewer — not waiting for a full LMS with auth and payments.

**Strategic Alignment:**
This MVP is the foundation for everything that follows: auth, paid tiers, certificates, and a CMS. Decisions made here (data model, routing, state management) should not block those phases — they should make them easier.

## 4. Assumptions

| Assumption | Validation Plan | Risk if Wrong |
|---|---|---|
| Non-technical users are comfortable with a browser-based experience and no login | Watch drop-off on first visit; gather feedback | May need to add a guest profile (name/email) earlier than planned |
| localStorage is durable enough for progress tracking in the short term | Monitor user feedback about lost progress | Add localStorage→DB sync or move to auth faster |
| Markdown is sufficient for lesson content quality | Review first seeded class with real content | May need image support, video embeds, or structured blocks sooner |
| One developer can seed content via SQL; no admin UI needed at launch | Track how often content changes are requested | Build a minimal admin page as soon as content changes exceed once/week |
| The `is_featured` flag on a single class is sufficient homepage personalization | Review homepage conversion (clicks to class start) | May need multiple featured classes or personalization logic |

## 6. User Interaction and Design

### User Flow Summary

```
Landing (/) ──→ See featured class hero
    │
    ├─ "Start Learning" ──→ /classes/:id (ClassDetail)
    │                           │
    │                           └─ "Start Class" (enroll) ──→ /classes/:id/lessons/1
    │                                                              │
    │                                                              └─ "Mark Complete" → Next Lesson → ... → /progress
    │
    └─ "Browse All Classes" ──→ /classes (Catalog)
                                    │
                                    └─ Click card ──→ /classes/:id
```

### Visual Design Tokens

| Token | Value |
|---|---|
| Background | `stone-50` / `stone-100` |
| Primary accent | `amber-600` (buttons, badges, active nav) |
| Text | `stone-700` (body), `stone-900` (headings) |
| Difficulty: Beginner | `bg-green-100 text-green-800` |
| Difficulty: Intermediate | `bg-amber-100 text-amber-800` |
| Difficulty: Advanced | `bg-red-100 text-red-800` |
| Progress bar fill | `bg-amber-500` on `bg-stone-200` track |
| Max-width container | `max-w-6xl mx-auto px-4 sm:px-6 lg:px-8` |

### UX Constraints
- No authentication UI — zero friction to start a class
- Lesson content rendered as markdown prose (`@tailwindcss/typography` `prose-stone`)
- Mobile-first: navbar collapses to hamburger on small screens
- Loading states required on all API-driven components (skeleton or spinner)
- Empty states required on catalog, progress dashboard, and lesson viewer (404 case)

## 8. What We're Not Doing (MVP Scope)

| Out of Scope | Future Consideration |
|---|---|
| User authentication / accounts | Phase 2 — likely Clerk or Supabase Auth |
| Video lesson content | Add `video_url` column to `lessons` and conditional iframe in LessonViewer |
| Admin CMS / content editing UI | Build when content change frequency > 1/week |
| Payments / paid tiers | Post-auth phase |
| Quizzes or assessments | After lesson completion mechanic is validated |
| Email collection / lead capture | Can add to registration flow before auth if needed |
| Certificates of completion | Post-auth (needs a persistent user identity) |
| Difficulty filtering on catalog | Easy Zustand addition — deprioritised for MVP |
| Search | Post-MVP |
| Multiple featured classes / personalization | Single `is_featured` flag is sufficient for now |
| Dark mode | Not aligned with warm earthy brand direction |
| Internationalization | Post-MVP |

## Tech Stack Reference

```
Frontend:   Vite + React + TypeScript + TailwindCSS + Zustand + React Router v6
Backend:    Node.js + Express + better-sqlite3
Database:   SQLite3 (catalog only — classes + lessons)
State:      Zustand with localStorage persistence (student progress)
Content:    Markdown (TEXT column in SQLite, rendered via react-markdown + remark-gfm)
Auth:       None (MVP)
```
